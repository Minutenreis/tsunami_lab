While Catch fully supports the traditional, x\+Unit, style of class-\/based fixtures containing test case methods this is not the preferred style.

Instead Catch provides a powerful mechanism for nesting test case sections within a test case. For a more detailed discussion see the \href{tutorial.md\#test-cases-and-sections}{\texttt{ tutorial}}.

Test cases and sections are very easy to use in practice\+:


\begin{DoxyItemize}
\item {\bfseries{T\+E\+S\+T\+\_\+\+C\+A\+SE($\ast$$\ast$ {\itshape test name} \mbox{[}, {\itshape tags} \mbox{]} $\ast$$\ast$)}}
\item {\bfseries{S\+E\+C\+T\+I\+ON($\ast$$\ast$ {\itshape section name} $\ast$$\ast$)}}
\end{DoxyItemize}

{\itshape test name} and {\itshape section name} are free form, quoted, strings. The optional {\itshape tags} argument is a quoted string containing one or more tags enclosed in square brackets. Tags are discussed below. Test names must be unique within the Catch executable.

For examples see the \href{tutorial.md\#top}{\texttt{ Tutorial}}\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md465}{}\doxysubsection{Tags}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md465}
Tags allow an arbitrary number of additional strings to be associated with a test case. Test cases can be selected (for running, or just for listing) by tag -\/ or even by an expression that combines several tags. At their most basic level they provide a simple way to group several related tests together.

As an example -\/ given the following test cases\+: \begin{DoxyVerb}TEST_CASE( "A", "[widget]" ) { /* ... */ }
TEST_CASE( "B", "[widget]" ) { /* ... */ }
TEST_CASE( "C", "[gadget]" ) { /* ... */ }
TEST_CASE( "D", "[widget][gadget]" ) { /* ... */ }
\end{DoxyVerb}


The tag expression, {\ttfamily \char`\"{}\mbox{[}widget\mbox{]}\char`\"{}} selects A, B \& D. {\ttfamily \char`\"{}\mbox{[}gadget\mbox{]}\char`\"{}} selects C \& D. {\ttfamily \char`\"{}\mbox{[}widget\mbox{]}\mbox{[}gadget\mbox{]}\char`\"{}} selects just D and {\ttfamily \char`\"{}\mbox{[}widget\mbox{]},\mbox{[}gadget\mbox{]}\char`\"{}} selects all four test cases.

For more detail on command line selection see \href{command-line.md\#specifying-which-tests-to-run}{\texttt{ the command line docs}}

Tag names are not case sensitive and can contain any A\+S\+C\+II characters. This means that tags {\ttfamily \mbox{[}tag with spaces\mbox{]}} and {\ttfamily \mbox{[}I said \char`\"{}good day\char`\"{}\mbox{]}} are both allowed tags and can be filtered on. Escapes are not supported however and {\ttfamily \mbox{[}\textbackslash{}\mbox{]}\mbox{]}} is not a valid tag.\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md466}{}\doxysubsubsection{Special Tags}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md466}
All tag names beginning with non-\/alphanumeric characters are reserved by Catch. Catch defines a number of \char`\"{}special\char`\"{} tags, which have meaning to the test runner itself. These special tags all begin with a symbol character. Following is a list of currently defined special tags and their meanings.


\begin{DoxyItemize}
\item {\ttfamily \mbox{[}!hide\mbox{]}} or {\ttfamily \mbox{[}.\mbox{]}} -\/ causes test cases to be skipped from the default list (i.\+e. when no test cases have been explicitly selected through tag expressions or name wildcards). The hide tag is often combined with another, user, tag (for example {\ttfamily \mbox{[}.\mbox{]}\mbox{[}integration\mbox{]}} -\/ so all integration tests are excluded from the default run but can be run by passing {\ttfamily \mbox{[}integration\mbox{]}} on the command line). As a short-\/cut you can combine these by simply prefixing your user tag with a {\ttfamily .} -\/ e.\+g. {\ttfamily \mbox{[}.integration\mbox{]}}. Because the hide tag has evolved to have several forms, all forms are added as tags if you use one of them.
\item {\ttfamily \mbox{[}!throws\mbox{]}} -\/ lets Catch know that this test is likely to throw an exception even if successful. This causes the test to be excluded when running with {\ttfamily -\/e} or {\ttfamily -\/-\/nothrow}.
\item {\ttfamily \mbox{[}!mayfail\mbox{]}} -\/ doesn\textquotesingle{}t fail the test if any given assertion fails (but still reports it). This can be useful to flag a work-\/in-\/progress, or a known issue that you don\textquotesingle{}t want to immediately fix but still want to track in your tests.
\item {\ttfamily \mbox{[}!shouldfail\mbox{]}} -\/ like {\ttfamily \mbox{[}!mayfail\mbox{]}} but {\itshape fails} the test if it {\itshape passes}. This can be useful if you want to be notified of accidental, or third-\/party, fixes.
\item {\ttfamily \mbox{[}!nonportable\mbox{]}} -\/ Indicates that behaviour may vary between platforms or compilers.
\item {\ttfamily \mbox{[}\#$<$filename$>$\mbox{]}} -\/ running with {\ttfamily -\/\#} or {\ttfamily -\/-\/filenames-\/as-\/tags} causes Catch to add the filename, prefixed with {\ttfamily \#} (and with any extension stripped), as a tag to all contained tests, e.\+g. tests in testfile.\+cpp would all be tagged {\ttfamily \mbox{[}\#testfile\mbox{]}}.
\item {\ttfamily \mbox{[}@$<$alias$>$\mbox{]}} -\/ tag aliases all begin with {\ttfamily @} (see below).
\item {\ttfamily \mbox{[}!benchmark\mbox{]}} -\/ this test case is actually a benchmark. This is an experimental feature, and currently has no documentation. If you want to try it out, look at {\ttfamily projects/\+Self\+Test/\+Benchmark.\+tests.\+cpp} for details.
\end{DoxyItemize}\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md467}{}\doxysubsection{Tag aliases}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md467}
Between tag expressions and wildcarded test names (as well as combinations of the two) quite complex patterns can be constructed to direct which test cases are run. If a complex pattern is used often it is convenient to be able to create an alias for the expression. This can be done, in code, using the following form\+: \begin{DoxyVerb}CATCH_REGISTER_TAG_ALIAS( <alias string>, <tag expression> )
\end{DoxyVerb}


Aliases must begin with the {\ttfamily @} character. An example of a tag alias is\+: \begin{DoxyVerb}CATCH_REGISTER_TAG_ALIAS( "[@nhf]", "[failing]~[.]" )
\end{DoxyVerb}


Now when {\ttfamily \mbox{[}@nhf\mbox{]}} is used on the command line this matches all tests that are tagged {\ttfamily \mbox{[}failing\mbox{]}}, but which are not also hidden.\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md468}{}\doxysubsection{B\+D\+D-\/style test cases}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md468}
In addition to Catch\textquotesingle{}s take on the classic style of test cases, Catch supports an alternative syntax that allow tests to be written as \char`\"{}executable specifications\char`\"{} (one of the early goals of \href{http://dannorth.net/introducing-bdd/}{\texttt{ Behaviour Driven Development}}). This set of macros map on to {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE}s and {\ttfamily S\+E\+C\+T\+I\+ON}s, with a little internal support to make them smoother to work with.


\begin{DoxyItemize}
\item {\bfseries{S\+C\+E\+N\+A\+R\+IO($\ast$$\ast$ {\itshape scenario name} \mbox{[}, {\itshape tags} \mbox{]} $\ast$$\ast$)}}
\end{DoxyItemize}

This macro maps onto {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE} and works in the same way, except that the test case name will be prefixed by \char`\"{}\+Scenario\+: \char`\"{}


\begin{DoxyItemize}
\item {\bfseries{G\+I\+V\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{W\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{T\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\end{DoxyItemize}

These macros map onto {\ttfamily S\+E\+C\+T\+I\+ON}s except that the section names are the \+\_\+something\+\_\+s prefixed by \char`\"{}given\+: \char`\"{}, \char`\"{}when\+: \char`\"{} or \char`\"{}then\+: \char`\"{} respectively.


\begin{DoxyItemize}
\item {\bfseries{A\+N\+D\+\_\+\+G\+I\+V\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{A\+N\+D\+\_\+\+W\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{A\+N\+D\+\_\+\+T\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\end{DoxyItemize}

Similar to {\ttfamily G\+I\+V\+EN}, {\ttfamily W\+H\+EN} and {\ttfamily T\+H\+EN} except that the prefixes start with \char`\"{}and \char`\"{}. These are used to chain {\ttfamily G\+I\+V\+EN}s, {\ttfamily W\+H\+EN}s and {\ttfamily T\+H\+EN}s together.

\begin{quote}
{\ttfamily A\+N\+D\+\_\+\+G\+I\+V\+EN} was \href{https://github.com/catchorg/Catch2/issues/1360}{\texttt{ introduced}} in Catch 2.\+4.\+0. \end{quote}


When any of these macros are used the console reporter recognises them and formats the test case header such that the Givens, Whens and Thens are aligned to aid readability.

Other than the additional prefixes and the formatting in the console reporter these macros behave exactly as {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE}s and {\ttfamily S\+E\+C\+T\+I\+ON}s. As such there is nothing enforcing the correct sequencing of these macros -\/ that\textquotesingle{}s up to the programmer!\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md469}{}\doxysubsection{Type parametrised test cases}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md469}
In addition to {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE}s, Catch2 also supports test cases parametrised by types, in the form of {\ttfamily T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+SE}, {\ttfamily T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+P\+R\+O\+D\+U\+C\+T\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+SE} and {\ttfamily T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+L\+I\+S\+T\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+SE}.


\begin{DoxyItemize}
\item {\bfseries{T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+SE($\ast$$\ast$ {\itshape test name} , {\itshape tags}, {\itshape type1}, {\itshape type2}, ..., {\itshape typen} $\ast$$\ast$)}}
\end{DoxyItemize}

\begin{quote}
\href{https://github.com/catchorg/Catch2/issues/1437}{\texttt{ Introduced}} in Catch 2.\+5.\+0. \end{quote}


{\itshape test name} and {\itshape tag} are exactly the same as they are in {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE}, with the difference that the tag string must be provided (however, it can be empty). {\itshape type1} through {\itshape typen} is the list of types for which this test case should run, and, inside the test code, the current type is available as the {\ttfamily Test\+Type} type.

Because of limitations of the C++ preprocessor, if you want to specify a type with multiple template parameters, you need to enclose it in parentheses, e.\+g. {\ttfamily std\+::map$<$int, std\+::string$>$} needs to be passed as {\ttfamily (std\+::map$<$int, std\+::string$>$)}.

\begin{DoxyItemize}
\item {\bfseries{T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+P\+R\+O\+D\+U\+C\+T\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+SE($\ast$$\ast$ {\itshape test name} , {\itshape tags}, ({\itshape template-\/type1}, {\itshape template-\/type2}, ..., {\itshape template-\/typen}), ({\itshape template-\/arg1}, {\itshape template-\/arg2}, ..., {\itshape template-\/argm}) $\ast$$\ast$)}}
\end{DoxyItemize}

\begin{quote}
\href{https://github.com/catchorg/Catch2/issues/1468}{\texttt{ Introduced}} in Catch 2.\+6.\+0. \end{quote}


{\itshape template-\/type1} through {\itshape template-\/typen} is list of template template types which should be combined with each of {\itshape template-\/arg1} through {\itshape template-\/argm}, resulting in {\itshape n $\ast$ m} test cases. Inside the test case, the resulting type is available under the name of {\ttfamily Test\+Type}.

To specify more than 1 type as a single {\itshape template-\/type} or {\itshape template-\/arg}, you must enclose the types in an additional set of parentheses, e.\+g. {\ttfamily ((int, float), (char, double))} specifies 2 template-\/args, each consisting of 2 concrete types ({\ttfamily int}, {\ttfamily float} and {\ttfamily char}, {\ttfamily double} respectively). You can also omit the outer set of parentheses if you specify only one type as the full set of either the {\itshape template-\/types}, or the {\itshape template-\/args}.

\DoxyCodeLine{\textcolor{keyword}{template}< \textcolor{keyword}{typename} T>}

\DoxyCodeLine{    \textcolor{keywordtype}{size\_t} size() \{}
\DoxyCodeLine{        \textcolor{keywordflow}{return} 0;}
\DoxyCodeLine{    \}}
\DoxyCodeLine{\};}
\DoxyCodeLine{}
\DoxyCodeLine{TEMPLATE\_PRODUCT\_TEST\_CASE(\textcolor{stringliteral}{"A Template product test case"}, \textcolor{stringliteral}{"[template][product]"}, (std::vector, Foo), (\textcolor{keywordtype}{int}, \textcolor{keywordtype}{float})) \{}
\DoxyCodeLine{    TestType x;}
\DoxyCodeLine{    REQUIRE(x.size() == 0);}
\DoxyCodeLine{\}}
\end{DoxyCode}

\item {\bfseries{T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+L\+I\+S\+T\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+SE($\ast$$\ast$ {\itshape test name}, {\itshape tags}, {\itshape type list} $\ast$$\ast$)}}

This allows you to reuse the {\itshape type list} in multiple test cases.

\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md470}{}\doxysubsection{Signature based parametrised test cases}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md470}
\begin{quote}
\href{https://github.com/catchorg/Catch2/issues/1609}{\texttt{ Introduced}} in Catch 2.\+8.\+0. \end{quote}


In addition to \href{\#type-parametrised-test-cases}{\texttt{ type parametrised test cases}} Catch2 also supports signature base parametrised test cases, in form of {\ttfamily T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+S\+E\+\_\+\+S\+IG} and {\ttfamily T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+P\+R\+O\+D\+U\+C\+T\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+S\+E\+\_\+\+S\+IG}. These test cases have similar syntax like \href{\#type-parametrised-test-cases}{\texttt{ type parametrised test cases}}, with one additional positional argument which specifies the signature.\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md471}{}\doxysubsubsection{Signature}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md471}
Signature has some strict rules for these tests cases to work properly\+:
\begin{DoxyItemize}
\item signature with multiple template parameters e.\+g. {\ttfamily typename T, size\+\_\+t \mbox{\hyperlink{structS}{S}}} must have this format in test case declaration {\ttfamily ((typename T, size\+\_\+t \mbox{\hyperlink{structS}{S}}), T, \mbox{\hyperlink{structS}{S}})}
\item signature with variadic template arguments e.\+g. {\ttfamily typename T, size\+\_\+t \mbox{\hyperlink{structS}{S}}, typename...Ts} must have this format in test case declaration {\ttfamily ((typename T, size\+\_\+t \mbox{\hyperlink{structS}{S}}, typename...Ts), T, \mbox{\hyperlink{structS}{S}}, Ts...)}
\item signature with single non type template parameter e.\+g. {\ttfamily int V} must have this format in test case declaration {\ttfamily ((int V), V)}
\item signature with single type template parameter e.\+g. {\ttfamily typename T} should not be used as it is in fact {\ttfamily T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+T\+E\+S\+T\+\_\+\+C\+A\+SE}
\end{DoxyItemize}

Currently Catch2 support up to 11 template parameters in signature\hypertarget{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md472}{}\doxysubsubsection{Examples}\label{md_submodules_Catch2_docs_test-cases-and-sections_autotoc_md472}
